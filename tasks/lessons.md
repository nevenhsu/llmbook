# Lessons Learned

## AI Agent Plan

- Use the real repo routes in plans and specs: `/admin/ai/agent-lab` for admin runtime, `/preview/ai-agent-lab` for mock preview.
- Keep the route split explicit: Preview uses mock fixtures, Admin uses live DB/API runtime data, and only the shared lab surface is unified.
- Use `Runtime` labels for live admin modes and `Preview` labels for mock modes so source-of-truth differences stay obvious.
- For manual task injection, keep the runtime contract canonical: reuse `inject_persona_tasks` RPC and do not add a direct insert path.
- Keep preview/admin control parity for lab UX. If Admin shows model selection or save actions, Preview should usually render the same controls with mock-backed behavior.
- For task saves, support both table-level `Save All` and row-level `Save`, implemented as serial single-candidate saves for clear per-row debugging.
- Treat successful task saves as terminal in the UI: `Save All` continues past failures, later retries skip saved rows, and row `Save` is disabled after success.
- Align mock JSON to the admin/runtime contract first. Preview-only states like toast, `saved`, or button disabled state belong in UI state, not canonical payloads.
- When a lab is meant to predict runtime behavior, do not rely only on shared helpers; extract a canonical shared trace/service and make runtime plus lab consume it directly.
- Protect stage format changes with contract tests on the shared trace so `Opportunities / Candidates / Tasks` changes fail in one place instead of drifting between runtime and lab.
- For UI spacing requests, do not infer direction from “adjust”; explicitly preserve or confirm whether spacing should become larger or smaller before refining the layout.
- In multi-table card UIs, add per-table titles only when a card actually contains multiple tables; do not add a redundant heading above a single table.
- Keep stage ownership aligned with stage names: `Opportunities` should stop at selected opportunities plus selection reasons, `Candidates` should own LLM-picked reference names plus per-reference reasons, and persona resolution should remain a later stage instead of being folded into candidate selection.
- For ai-agent intake contracts, prefer stable local keys in all LLM JSON stages, keep `recipient_persona_id` and `persona_id` app-owned via snapshot/DB resolution, keep candidate-stage output reason-free for format consistency, and make resolved-persona output explicitly carry `{ reference_name, persona_id, status }` rows keyed by local opportunity.
- Keep notification and public opportunity selectors structurally aligned: both should emit only selected local keys under `selected_opportunities`, while any `reason` needed for admin inspection should come from a separate admin-only artifact instead of the canonical runtime JSON.
- For staged LLM prompts shown in admin/preview, do not describe outputs loosely as “return keys only”; embed the canonical top-level JSON shape directly in the assembled prompt so `selected_opportunities` / `candidate_selections` stay explicit and testable.
- Do not reuse default `comment` / `post` output constraints for non-action LLM stages like opportunities or candidates. Those stages must override `output_constraints` explicitly, or the admin prompt modal will incorrectly show markdown/image task-generation contracts.
- Keep intake-stage prompts separated by business logic, not just by output schema. `Opportunities` and `Candidates` must each have their own prompt architecture and block structure because one selects tasks from snapshot data while the other selects reference names from already-selected tasks.
- When the lab already has a canonical trace at initial render, keep the derived stage prompt previews in initial mode state so `Show Prompt` is immediately available; do not null them out just because the stage status is still `idle`.
- Mock preview data should help differentiate stage behavior. When multiple selected opportunities are shown in `Candidates`, avoid reusing the exact same mock `selected_references` set for every row unless the product logic explicitly requires it.
- In `AI Agent Lab`, `group index` and `batch size` cannot be cosmetic local state. They must flow into the canonical candidate-stage input, refresh the selected reference list immediately, and invalidate downstream resolved/tasks rows that were derived from the old reference batch.
- For selector-style LLM stages, the assembled prompt must show both the required JSON schema and the actual decision criteria. If the prompt only describes output format, the admin lab cannot explain why an opportunity or reference should be selected.
- For staged JSON outputs that are array-shaped, `required_output_json` should show a multi-item example and explicit array/object rules, not just a single-row sample. Admin preview needs to make the full payload shape obvious.
- For `AI Agent Lab`, do not keep split input/result tables once the stage contract becomes row-complete. If the LLM output can be merged onto the source rows deterministically, prefer one paginated table over two synchronized tables.
- Do not reuse “selected keys” wording after moving to probability-based opportunity selection. When the app derives selection from `probability > 0.5`, the prompt, table columns, and data modal labels must all describe probability output explicitly.
- When a prompt modal is meant to show only the assembled prompt for a stage, hide empty supporting sections entirely instead of rendering `null` JSON blocks. Otherwise the UI still reads like it expects extra payload sections.
- For admin/debug data modals, distinguish parsed LLM output from internal trace wrappers. If the user asked for “output data”, do not pass through `promptPreview` / `actualModelPayload` or other stage diagnostics unless explicitly requested.
- Local prompt keys must never leak into persistence-facing fields. If a runtime/task payload needs both UI-local identity and DB identity, carry `opportunityKey` and real `sourceId` as separate fields instead of overloading one property.
- When prompt previews are pre-seeded for an idle stage, keep the prompt active without pre-populating run results. Initial table rows must still reflect the idle state, not a synthetic successful execution.
- If notification mode is supposed to disable a control, disable the entry-point button as well as any modal internals. A disabled field inside an opened modal is not enough.
- For merged selection tables, verify the data merge against the full source list, not just selected rows. Otherwise the UI can show the right columns but still violate the approved table contract.
- In idle tables, `unknown` must render as `-`, not as a negative decision. If a run-dependent field has no value yet, do not coerce it into `false`/`No`.
- If a source mode already has deterministic upstream inputs on first render, prefill the corresponding table from canonical trace data instead of showing an empty placeholder. Empty tables imply missing data, not idle state.
- Preview mock data must be internally consistent with the visible controls. Do not display `batch size`, `group index`, or `total reference count` values that the seeded mock reference library and snapshot cannot actually produce.
- For seeded pre-run candidate tables, separate “reference group exists” from “opportunity assignment exists”. Initial rows may show reference/persona data, but `Opportunity Key` must stay empty until a run produces a real mapping.
- If a mock selection table is meant to demonstrate positive and negative cases, seed at least one explicit negative row in the default preview. Otherwise the preview cannot validate the `false` path.
- When downstream tables are derived from an upstream selected table, match the downstream row order to the upstream visible selection order. A correct payload in a different order still feels wrong in admin UX.
- If downstream task materialization filters inactive personas, expose persona active/inactive status directly in the upstream `Candidates` table. Otherwise the UI reads like mock data drift instead of a deliberate filter.
- When a user asks for a status field as a table column, keep it as an explicit column. Do not hide it inside another cell, even if the data is available there.
- For admin/preview data modals, show the persistence/input payload when the user asks for “data used to save”. Do not default to response payloads or full UI state objects.
- For status columns in dense admin tables, prefer explicit text labels plus simple indicators over tiny decorative pills. If the label cannot be read at a glance, the UI fails even when the underlying data is correct.
- The same rule applies to task/save lifecycle columns: never leave important state as raw lowercase strings when a table needs fast scanning. Use explicit labels plus consistent visual indicators.
- When a preview is meant to explain a filtered-out downstream row, seed that exact edge case in the default mock data. A visible inactive-selected candidate is better than relying on users to infer the rule from hidden fixture logic.
- When adjacent admin tables show the same domain entity, keep the entity cell UI consistent across them. If `Candidates` uses avatar/display-name/username for personas, `Tasks` should not collapse back to plain text.
- When a debug modal is meant to explain persistence, show the closest persistence-facing shape you can. HTTP wrapper bodies obscure the real DB/RPC contract and make valid payload keys look accidental.
