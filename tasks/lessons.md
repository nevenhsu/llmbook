# Lessons Learned

## AI Agent Plan

- Use the real repo routes in plans and specs: `/admin/ai/agent-lab` for admin runtime, `/preview/ai-agent-lab` for mock preview.
- Keep the route split explicit: Preview uses mock fixtures, Admin uses live DB/API runtime data, and only the shared lab surface is unified.
- Use `Runtime` labels for live admin modes and `Preview` labels for mock modes so source-of-truth differences stay obvious.
- For manual task injection, keep the runtime contract canonical: reuse `inject_persona_tasks` RPC and do not add a direct insert path.
- Keep preview/admin control parity for lab UX. If Admin shows model selection or save actions, Preview should usually render the same controls with mock-backed behavior.
- For task saves, support both table-level `Save All` and row-level `Save`, implemented as serial single-candidate saves for clear per-row debugging.
- Treat successful task saves as terminal in the UI: `Save All` continues past failures, later retries skip saved rows, and row `Save` is disabled after success.
- Align mock JSON to the admin/runtime contract first. Preview-only states like toast, `saved`, or button disabled state belong in UI state, not canonical payloads.
- When a lab is meant to predict runtime behavior, do not rely only on shared helpers; extract a canonical shared trace/service and make runtime plus lab consume it directly.
- Protect stage format changes with contract tests on the shared trace so `Opportunities / Candidates / Tasks` changes fail in one place instead of drifting between runtime and lab.
- Keep stage ownership aligned with stage names: `Opportunities` should stop at selected opportunities plus selection reasons, `Candidates` should own LLM-picked reference names plus per-reference reasons, and persona resolution should remain a later stage instead of being folded into candidate selection.
- For ai-agent intake contracts, prefer stable local keys in all LLM JSON stages, keep `recipient_persona_id` and `persona_id` app-owned via snapshot/DB resolution, keep candidate-stage output reason-free for format consistency, and make resolved-persona output explicitly carry `{ reference_name, persona_id, status }` rows keyed by local opportunity.
- Keep notification and public opportunity selectors structurally aligned: both should emit only selected local keys under `selected_opportunities`, while any `reason` needed for admin inspection should come from a separate admin-only artifact instead of the canonical runtime JSON.
- For staged LLM prompts shown in admin/preview, do not describe outputs loosely as “return keys only”; embed the canonical top-level JSON shape directly in the assembled prompt so `selected_opportunities` / `candidate_selections` stay explicit and testable.
- Do not reuse default `comment` / `post` output constraints for non-action LLM stages like opportunities or candidates. Those stages must override `output_constraints` explicitly, or the admin prompt modal will incorrectly show markdown/image task-generation contracts.
- Keep intake-stage prompts separated by business logic, not just by output schema. `Opportunities` and `Candidates` must each have their own prompt architecture and block structure because one selects tasks from snapshot data while the other selects reference names from already-selected tasks.
- When the lab already has a canonical trace at initial render, keep the derived stage prompt previews in initial mode state so `Show Prompt` is immediately available; do not null them out just because the stage status is still `idle`.
- Mock preview data should help differentiate stage behavior. When multiple selected opportunities are shown in `Candidates`, avoid reusing the exact same mock `selected_references` set for every row unless the product logic explicitly requires it.
- In `AI Agent Lab`, `group index` and `batch size` cannot be cosmetic local state. They must flow into the canonical candidate-stage input, refresh the selected reference list immediately, and invalidate downstream resolved/tasks rows that were derived from the old reference batch.
- Local prompt keys must never leak into persistence-facing fields. If a runtime/task payload needs both UI-local identity and DB identity, carry `opportunityKey` and real `sourceId` as separate fields instead of overloading one property.
- In idle tables, `unknown` must render as `-`, not as a negative decision. If a run-dependent field has no value yet, do not coerce it into `false`/`No`.
- If downstream task materialization filters inactive personas, expose persona active/inactive status directly in the upstream `Candidates` table. Otherwise the UI reads like mock data drift instead of a deliberate filter.
- If multiple selected references can resolve to the same persona, downstream task materialization must dedupe on `opportunity + persona`, and the default preview should include that edge case so the behavior is visible instead of implicit.
- For public opportunities, a bare post title is not enough to express why posting is appropriate. Include board context directly in the summary/content string so both the table UI and the prompt make the target board legible.
- If a payload wrapper adds no real domain boundary and only hides direct target ids, flatten it. Runtime/debug contracts should not carry extra nested objects like `notificationTarget` when the values are always consumed as plain payload fields.
- When one path carries explicit target ids in payload, the sibling path should not leave them null by convention. Keep public and notification payload contracts structurally aligned by carrying board/post/comment target ids wherever the app already knows them.
- When an upstream stage is rerun, clear any downstream assignment/output state that came from the previous run. Keeping old candidate keys or old output visible after a new selector run creates false state even if the new run has not reached that stage yet.
- Do not model rotating batch progress with a single “last processed group” scalar when new opportunities can arrive after the cursor has already advanced. Persist per-opportunity per-group progress, then apply stop rules like `matched_persona_count >= 3` using cumulative unique resolved personas, not raw reference-name counts.
- Keep public and notification runtime flows split at the candidate stage. Notification rows may share the same `ai_opps` ingestion and opportunity-scoring pipeline, but once `selected = true`, they should bypass candidate-group logic entirely because `recipient_persona_id` is already deterministic.
- Do not reuse public stop rules for notification opportunities. Public rows may stop after accumulating enough unique personas, but notification rows should become terminal after one downstream processing pass and must be excluded from future runtime queries explicitly.
- Keep admin manual flows separate from background runtime progress. Admin surfaces may share `ai_opps` and `persona_tasks`, but manual task saves must not silently advance runtime group cursors or claim automatic candidate-group processing on behalf of the app.
- When a field represents accumulated opportunity coverage, encode the monotonic rule explicitly. `matched_persona_count` should be a cumulative per-opportunity count that only increases on new unique persona matches and is not reset by group rotation or admin reruns.
- Keep operator/manual flows independent from background runtime cursors unless the product explicitly says otherwise. Sharing source tables does not imply sharing rotation state like `public_candidate_group_index`.
- When you persist upstream opportunity rows, store every downstream target field needed to rebuild runtime task payloads. If `ai_opps` drops fields like `parent_comment_id`, notification context/type, or source event time, persisted rows cannot safely outlive the original snapshot.
- A persisted runtime pipeline must query persisted eligibility after ingest/scoring, not treat an empty current snapshot as terminal. Otherwise selected rows already stored in `ai_opps` can get stranded even though they are still eligible for downstream processing.
- Preview/demo selection builders are not valid runtime services. Keep fixture helpers like prompt/selection preview builders out of runtime/admin execute paths; runtime must read persisted state, config, reference sources, and persona tables directly.
- When the product direction is “new flow only,” do not leave read-only admin/preview surfaces on legacy artifact builders just because they are not executing writes. Trace/demo helpers left in visible product UI still create contract drift and must be removed alongside runtime entrypoints.
- When the user says old flow/code/tests/docs must be gone, do a full-repo symbol/doc sweep after the runtime migration. Deleting execute-path usage is not enough if unused helpers, stale tests, or plan text still preserve the old contract.
- After migrating runtime/admin to a staged LLM contract, remove preview-side selector heuristics too. Mock surfaces may stay fixture-backed, but they should read explicit fixture outputs instead of silently recomputing probabilities or speaker picks with old helper logic.
- For staged selector LLMs that operate on variable snapshot sizes, bake batching and omission-repair into the shared service itself. Keep batches small (for this flow: 10 rows), cap the public cycle deterministically, and rerun only missing local-key subsets once instead of resending the whole original batch.
- In public candidate selection, the non-LLM fallback belongs after reference-name-to-persona resolution. If the LLM returns no names or only unusable personas, do not rerun the model; pick one random persona from the current batch's active, non-duplicate personas instead.
- When shortening a canonical staged-JSON key for prompt readability, rename it everywhere in one pass: prompt examples, parser exact-key validation, admin modal copy, fixture output, and tests. Leaving the old key in any one layer creates contract drift immediately.
- If an LLM stage is already batched for drift/token safety and its output is persisted, commit each completed batch before starting the next one. Otherwise a crash midway through the cycle throws away already-finished work and forces a full rerun.
- Apply the same incremental-persistence rule to downstream candidate selection, not just opportunity scoring. If candidates run in 10-opportunity batches, each batch should record group progress, insert tasks, and update counts before the next batch starts.
- When an admin lab action is meant to materialize real tasks, do not leave the UI in an `idle` save state after the backend insert already happened. Batch-save during the run and return task rows with real success/failed states.
- For rerun behavior in admin/preview labs, preserve completed rows in the table but remove them from the next stage input. Re-running a stage should target unfinished or failed opportunities, not silently resend already-successful work to the model.
- For preview/mock stage surfaces, do not assume persisted selector values live inside `selectorInput.opportunityLookup`. JSON fixtures may carry `probability/selected` only on snapshot source items, so run-state builders must merge item metadata back into lookup rows before rendering post-run outputs.
- If notification recipient eligibility is deterministic, keep it out of the LLM stage entirely. Filter inactive recipient personas before opportunities scoring, persist the exclusion onto `ai_opps`, and prevent the same inactive notification row from reappearing in later sync/task cycles.
- Do not rely on “this only happens once in today’s code” for runtime LLM repair paths. Make retry, schema-repair, quality-repair, and subset-rerun limits explicit constants so config drift or future refactors cannot turn a bounded recovery path into a hang.
- For admin pages running staged LLM work behind Vercel limits, batch on the client before each API call. Runtime/background services may keep their own server-side batching, but admin page requests should not hand the whole 100-row workload to one server invocation.
- For manual admin lab runs, one click should mean one 10-row API batch only. Do not quietly auto-loop the next batches on the client after the first batch finishes.
- When a server batch already inserts `persona_tasks`, return explicit task outcomes keyed to the logical row (`opportunity`, `persona`) instead of making the client infer save state from `insertedTasks` and RPC result fragments.
- If admin batching is meant to feel progressive, do not stop at batch-level persistence. Push each completed batch back into page state immediately so `Candidates` assignments and `Tasks` save states reflect real progress before the final batch resolves.
- Apply the same progressive-state rule to upstream stages too. If `Opportunities` scoring is batched, refresh the table after each completed batch so probabilities/selection state do not lag behind persisted `ai_opps` updates.
- For admin tables that communicate decision state, do not leave binary fields as bare text like `Yes / No / -`. Use the same explicit status treatment as other lifecycle columns (`Selected / Not Selected / Pending` with visible indicators) so scanability stays consistent across the lab.
- When documenting or implementing runtime cycle limits, check both public and notification paths explicitly. If opportunities scoring is cycle-limited for one intake kind, confirm whether the sibling intake kind should share that same cap instead of assuming it is public-only.
- Do not couple admin-page manual batch controls to runtime config keys unless the product explicitly says so. A page-local “process N rows per click” control should not write back to `ai_agent_config.selector_reference_batch_size` or alter runtime behavior.
- For public opportunity matching, `matched_persona_count` must follow successfully inserted unique personas only. Do not increment it from raw LLM selections or pre-insert resolved personas, or reruns/duplicates will corrupt the cumulative stop rule.
- Distinguish automatic runtime cooldown from explicit operator control. Active cooldown may block scheduled cycles, but a manual `run_phase_a` should still be allowed whenever there is no active lease/running cycle.
- Manual `run_phase_a` must not reset or extend the orchestrator cooldown timer on success. It bypasses cooldown for dispatching, but it should leave the existing cooldown schedule unchanged so operator requests do not silently reschedule automatic runtime pacing.
- Do not treat `orchestrator_runtime_state` row availability, lease state, or cooldown state as proof that the runtime app is online. Operator controls that dispatch background runtime work need a separate runtime-app heartbeat and panel polling path, or the UI will offer actions that no runner can actually consume.
