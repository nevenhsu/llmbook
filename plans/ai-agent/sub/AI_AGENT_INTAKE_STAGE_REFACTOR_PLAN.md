# AI Agent Intake Stage Refactor Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Realign ai-agent intake so `Opportunities`, `Candidates`, `Resolved Personas`, and `Resolve Tasks` each own one clear responsibility, with one canonical local-key JSON contract shared by runtime, admin, preview, and prompt assembly.

**Architecture:** Split the current selector-driven intake flow into two LLM stages for public work: `Public Opportunities` selects opportunities from the source snapshot, and `Public Candidates` selects reference names for each selected opportunity. Keep notification intake structurally aligned by having `Notification Opportunities` also emit only selected local keys, while `recipient_persona_id`, `persona_id`, and source-ID mapping remain deterministic app-owned data. Admin-only inspection artifacts may carry `reason`, but canonical runtime JSON must stay minimal and reason-free wherever possible to reduce token usage.

**Tech Stack:** Next.js App Router, TypeScript, `src/lib/ai/agent/intake/*` shared contracts, admin agent lab surface, runtime/admin API routes, Vitest, existing prompt-runtime helpers.

## Contract Summary

### Canonical Stage Order

1. `Notification Opportunities`
2. `Public Opportunities`
3. `Public Candidates`
4. `Resolved Personas`
5. `Resolve Tasks`

### Ownership Rules

- LLM stages only consume and emit stable local keys such as `opportunity_key`.
- `recipient_persona_id` is snapshot-owned metadata for notification flows and must never be generated by LLM output.
- `persona_id` is DB/app-owned metadata derived from `persona_reference_sources` and `personas`; LLM stages must never emit it.
- `reason` is an admin preview/debug artifact, not canonical runtime JSON.
- `opportunity_key` must map deterministically to the real source row context needed later:
  - notification opportunity -> notification row + recipient persona + target linkage
  - public post opportunity -> board/post linkage
  - public reply opportunity -> board/post/comment linkage

### Local Key Strategy

- `opportunity_key` is a prompt-local key, not a persistence key.
- Runtime/admin code must maintain a deterministic lookup from `opportunity_key` back to the real source context.
- The lookup record for each key must retain the fields needed downstream:
  - notification opportunity -> `notification_id`, `recipient_persona_id`, target linkage
  - public post opportunity -> `board_id`, `post_id`
  - public reply opportunity -> `board_id`, `post_id`, `comment_id`
- LLM stages may see `opportunity_key`, source type, and summary/context, but must not see or emit raw DB identifiers unless explicitly required by a later approved contract.
- `opportunity_key` generation must be deterministic within a snapshot:
  - sort snapshot rows by a fixed rule
  - assign compact keys such as `N01`, `N02`, `O01`, `O02`
  - preserve the lookup table for all later stages in the same request/preview flow
- `Resolve Tasks` must always convert `opportunity_key` back through the lookup map before building `persona_tasks` candidates or dedupe/cooldown keys.

### Canonical LLM JSON Shapes

`Notification Opportunities` and `Public Opportunities` should converge on the same runtime shape:

```json
{
  "selected_opportunities": [
    {
      "opportunity_key": "N07"
    },
    {
      "opportunity_key": "O12"
    }
  ]
}
```

`Public Candidates` runtime/admin canonical shape:

```json
{
  "candidate_selections": [
    {
      "opportunity_key": "O07",
      "selected_references": ["Lu Xun", "Yu Hua"]
    }
  ]
}
```

`Resolved Personas` app-owned deterministic shape:

```json
{
  "resolved_candidates": [
    {
      "opportunity_key": "O07",
      "persona_ids": [
        {
          "reference_name": "Lu Xun",
          "persona_id": "persona_1",
          "status": "active"
        },
        {
          "reference_name": "Yu Hua",
          "persona_id": "persona_8",
          "status": "active"
        }
      ]
    }
  ]
}
```

`Resolve Tasks` app-owned deterministic shape must materialize canonical `persona_tasks` candidates from `(opportunity_key, persona_id)` pairs and include the fields needed by `inject_persona_tasks`.

### Admin-Only Artifacts

- Admin preview may render a sibling artifact that includes `reason` for:
  - notification opportunity selection
  - public opportunity selection
- `Public Candidates` does not include reason even for admin mode.
- Admin-only artifacts must not become the runtime JSON contract or prompt-repair target.
